§ 01 · Design Philosophy

设计哲学：为什么需要 CLADA

传统 AI 编程的失败模式有三个根源：上下文熵增（长期迭代后 AI 忘记早期决策）、单点权威缺失（没有任何机制阻止 AI 改变自己的规则）、以及验证与执行混同（同一个 Agent 既写代码又判断代码好坏）。

CLADA 的核心思想是将这三个问题分别用三个机制解决：三级记忆解决熵增、宪法文件（Contract + ADR）解决权威缺失、双 Agent 分权解决验证混同。

⚖️
三权分立
Owner（最高裁决）、Executor（执行权）、Verifier（审计权）三者权责边界由物理机制强制执行，而非依赖 Prompt 约定。
Constitutional
🧠
文档即共享内存
Agent 之间没有直接对话。所有信息交换通过结构化文件进行，每个文件格式都经过严格定义，防止语义漂移。
Memory-First
🔄
状态机驱动
系统在八个明确定义的状态之间跳转。每次状态转移都有物理触发条件，Gateway 是唯一的状态时钟。
Deterministic
🔒
物理隔离
Executor 的文件访问权限由 Gateway 在操作系统层级代理拦截。Verifier 的测试运行在 Docker 容器内，测试代码无法影响宿主机。
Hardware-Enforced

§ 03 · Gateway

Gateway 控制器详细设计

Gateway 是整个系统的神经中枢，以 orchestrator.py 形式运行。它不是简单的脚本，而是一个多线程进程管理器。

核心组件

PTY Wrapper
通过 pty.openpty() 启动 Agent 进程。Agent 认为自己在与标准终端对话，Gateway 截获所有 I/O 流。支持 SIGSTOP / SIGCONT 物理挂起。
Pattern Monitor 线程
持续嗅探 Executor 的 stdout。正则匹配触发词（[REQ_REVIEW]、[DONE]、[B_PLAN]）后立即发出状态切换指令。
文件访问代理
通过 LD_PRELOAD 或 ptrace 拦截 Executor 进程的文件系统调用。禁止读取 .env、secrets/、docs/decisions/（写操作）。
Heartbeat 守护线程
30 秒计时器。stdout 静默超时后向 PTY 发送 #: heartbeat\n 探针。若无响应，发送 show current status 强制 Executor 输出 TRACE。

current_state.json — 全局状态文件

§ 04 · State Machine

完整状态机定义

八个状态

IDLE
系统等待 Owner 指令。无 Agent 持有文件写权限。
BOOTSTRAP
创世阶段。Verifier 以"格式化工具模式"生成初始 Contract 和 DR-001。Owner 必须高强度介入确认。
PROPOSING
Verifier 接管。读取 L2 文档，与 Owner 协商，输出 current_spec.md。
EXECUTING
Executor 持有代码写权限。每修改 3 个文件输出一次 [TRACE]。Heartbeat 守护激活。
SUSPENDED
Executor 发出 [REQ_REVIEW]。Gateway 发送 SIGSTOP，检查 TCP 连接状态，准备切换至 Verifier。
ARBITRATING
Verifier 接管。必须引用 DR-xxx 做裁决，将决策追加注入 Executor STDIN。
AUDITING
src/ 目录 chmod 555 锁定。Verifier 在 Docker 容器内运行测试，生成 audit_report.json。
PENDING_COMMIT
审计通过，等待 Owner 执行 /merge 或 /reject。若含 [B_PLAN] 则不可进入 autopilot。

§ 05 · Communication Protocols

Agent 间通讯协议

ARBITRATING → EXECUTING：Append Injection

Verifier 的裁决通过 Gateway 以 SYSTEM_NOTIFICATION 格式直接写入 Executor 的 PTY STDIN，伪装成 Owner 指令。Executor 不需要读取第三方文件，在会话流中直接获取最新指令。

注入格式（写入 Executor PTY STDIN）
[SYSTEM_NOTIFICATION]: Verifier has resolved your query.
Decision: {裁决内容}
Referenced ADR: DR-024
Constraint: This decision is binding. Do not deviate.
Please integrate this decision and continue your current task.

EXECUTING → SUSPENDED：q_waiting.json

.comm/q_waiting.json
{
  "iteration_id": "IT-042",
  "question_id": "Q-007",
  "type": "interface_ambiguity",   // 歧义类型
  "context": "Spec 未定义跨模块接口的错误传递方式",
  "affected_files": ["src/auth.ts", "src/session.ts"],
  "quota_used": 3,
  "timestamp": 1716880200
}

[TRACE] 输出规范

Executor 每修改 3 个文件输出一次
[TRACE] IT-042 | Modified: src/auth.ts, src/db.ts, src/session.ts
[TRACE] Status: Implementing login endpoint per ASSERT-01
[TRACE] Next: Write unit test for Argon2id hash verification

消费路径：Gateway 实时将 TRACE 发送给 Summarizer（轻量级模型），生成 current_progress.md，供 Owner 实时查看。Summarizer 输出严禁进入 Executor 或 Verifier 的上下文。

[B_PLAN] 标签协议

触发条件 Executor Quota 耗尽但任务未完成时，必须在下一次代码提交中附加 [B_PLAN] 标签，表明此次决策为"无Spec依据的主观判断"。 
// 代码注释中标注
// [B_PLAN]: Session expiry set to 24h — no spec guidance, owner must confirm

// TRACE 中也必须出现
[TRACE] [B_PLAN] IT-042 | Decision without spec: session TTL = 86400s

审计后果：Verifier 发现 [B_PLAN] 后，即便测试全部通过，审计结论为 PASS_WITH_RISK，强制进入 WAITING_FOR_OWNER，禁止 autopilot 合并。

§ 06 · Contract Specification

contract.json 完整规范

Contract 是系统的"机器可读宪法"，优先级高于一切自然语言文档。Verifier 以此为审计标准，Executor 以此为实现边界。

字段分类

contract_id Hard
string
格式 CNT-xxx，唯一标识。版本升级时创建新 CNT，旧 CNT 标记 superseded_by。
version Hard
semver
语义化版本。Hard Fields 变更必须升 major，Soft Fields 变更升 minor。
scope.modules Hard
string[]
本 Contract 覆盖的模块列表。Executor 不得修改范围外的模块而不创建新 Contract。
scope.interfaces Hard
string[]
涉及的 API 端点或函数签名。格式：METHOD /path 或 FunctionName(args): ReturnType。
constraints.strict_types Hard
boolean
是否强制严格类型。影响 Verifier 的类型检查策略。
constraints.max_latency_ms Hard
number
性能硬指标。Verifier 的测试脚本中必须包含延迟断言。
constraints.allowed_dependencies Hard
string[]
白名单依赖库。Executor 引入白名单外依赖时 Gateway 拦截并报错。
hard_assertions Hard
Assertion[]
每条断言包含 id、description、check_script（可执行验证脚本路径）。Verifier 直接运行脚本，不做主观判断。
soft_recommendations Soft
Rec[]
编码风格建议，不影响审计结论。Owner 在双检锁阶段自行选择。
superseded_by Soft
string?
废弃时填写继任 CNT 编号。RAG 系统据此过滤过期 Contract。

完整示例

docs/spec/contract.json
{
  "contract_id": "CNT-001",
  "version": "1.0.0",
  "created_by": "bootstrap-dual-lock",
  "bootstrap_warning": "未经历史 ADR 审计，由 Owner 手动确认",
  "scope": {
    "modules": ["auth", "database"],
    "interfaces": ["POST /v1/login", "POST /v1/logout"]
  },
  "constraints": {
    "strict_types": true,
    "max_latency_ms": 200,
    "allowed_dependencies": ["zod", "argon2", "jsonwebtoken"]
  },
  "hard_assertions": [
    {
      "id": "ASSERT-01",
      "description": "密码必须在存储前用 Argon2id 哈希处理",
      "check_script": "npm run test:security"
    },
    {
      "id": "ASSERT-02",
      "description": "登录接口 P95 延迟必须低于 200ms",
      "check_script": "npm run test:perf -- --assert-p95=200"
    }
  ],
  "soft_recommendations": [
    {
      "id": "REC-01",

    }
  ],

Contract 版本演进规则

当 Contract 从 1.0.0 升级时，旧 CNT 不删除，标记 superseded_by: "CNT-002"。历史 DR 中引用旧 CNT 编号的记录保持原样（历史真实性），但 RAG 检索时通过 L2 index.json 的 superseded 字段自动过滤，不向 Agent 提供过期 Contract 内容。

DR（Decision Record）是系统的"判例法库"。所有重大决策必须记录，Verifier 在仲裁时必须引用 DR 而非凭空判断。

Front-matter 字段（机器可读，RAG 过滤依据）

DR-xxx

title Hard
string

status Hard
enum

superseded_by Hard

YYYY-MM-DD

string[]

## Context
初始方案使用 Claude Opus，但长期运行的 Token 成本过高，在 100 天迭代周期内不可持续。

## Decision
切换至 DeepSeek V4-Pro 通过 CLI Bridge 接入作为 Executor 模型。

## Rationale
V4-Pro 在编码基准测试中达到可比分数，成本降低约 90%，对长周期 Ralph Loop 至关重要。

## Trade-offs
- **放弃**：Claude Opus 在复杂推理任务中的优势
- **获得**：每月约 $500 的成本节省，可持续运行能力

## Consequences
- ✅ 总 Token 成本下降 90%
- ⚠️ 复杂推理任务延迟轻微上升
- ❌ 部分多语言任务质量略有下降

## Verification
- Gateway 仪表盘监控 Token 用量
- 每次 Audit 运行 `npm run test:audit` 验证编码准确性

§ 09 · Bootstrap

Bootstrap 引导流程（clada init）

Bootstrap 是 CLADA 唯一要求 Owner 高强度介入的阶段。此阶段生成的 Contract 和 ADR 是整个项目的"宪法创世"，必须经过双检锁确认后才能进入稳态。

⚠ Bootstrap 警告 此阶段 Verifier 处于"格式化工具模式"，没有历史 ADR 可参考。生成的 contract.json 必须由 Owner 像审计法律合同一样仔细审阅。系统会在 Contract 上自动标注 bootstrap_warning 字段，直到经过至少一次完整 Iteration 验证后才移除。

双检锁机制

Bootstrap 阶段由两个不同模型分别生成 Contract，Gateway 进行字段级比对，Owner 只仲裁差异点。

Step 1 Owner 输入 Goal 文本 ↓ Step 2 Model A (Claude) 生成 contract_a.json Model B (GPT-4o) 生成 contract_b.json ↓ 并行 Step 3 Gateway 执行 Hard Fields 字段级 Key-Value 比对 ↓ Step 4a Hard Fields 不一致 → 标红，Owner 必须手动选择 Step 4b Soft Fields 不一致 → 标黄，Owner 可选择或忽略 Step 4c 所有字段一致 → 自动通过 ↓ Step 5 Owner 点击 Confirm → contract.json 状态从"流体"变为"晶体" ↓ Step 6 系统写入 DR-001.md（技术选型决策） 生成初始 L2 index.json 标注 bootstrap_warning，进入 IDLE

冷启动 RAG 处理

若 L3 为空（新项目），Gateway 扫描现有代码库生成 Repo Map，作为第一份 architecture.md 写入 L2。若代码库也为空，则 architecture.md 仅包含 Goal 和技术栈选型，后续每次 Iteration 增量追加。

§ 10 · Isolation

物理隔离机制详解

CLADA 的隔离分两个独立防御层，针对不同的安全目标，适用对象不同。

防御层适用对象防御目标实现方式
宿主机文件代理 Executor 逻辑泄密（读取 secrets、修改 docs） Gateway 通过 LD_PRELOAD 或 ptrace 拦截文件系统调用，返回 EACCES
Docker 容器隔离 Verifier（测试运行时） 执行风险（破坏性测试代码） Docker DIND + iptables 白名单网络 + 只读 Volume 挂载

AUDITING 阶段完整隔离流程

Gateway 执行序列
# 1. 进入 AUDITING 前锁定宿主机源码目录
chmod -R 555 ./src      # Read & Execute only，剥夺写权限

# 2. 启动 fswatch 守护（宿主机层面监控）
fswatch -r ./src | while read f; do
  echo "[SECURITY] Unauthorized write attempt: $f"
  kill -9 $EXECUTOR_PID   # 立即杀死违规进程
done &

# 3. 拉起 Docker 容器（只读挂载源码）
docker run --rm \
  -v $(pwd)/src:/app/src:ro \         # 只读挂载
  -v $(pwd)/test_results:/results \   # 可写输出目录
  --network=clada-test-net \           # 隔离网络（仅 LocalStack）
  clada-verifier:latest \
  npm run test:all

# 4. 测试完成后恢复权限
chmod -R 755 ./src
docker rm -f clada-verifier-$(ITERATION_ID)
⚠ 技术风险：fswatch 与 Docker Volume fswatch 监控宿主机 inode。容器内通过 Volume 映射的写操作是否能被宿主机 fswatch 捕获，取决于 Volume 挂载方式（bind mount vs overlay）。在 bind mount 模式下 inode 共享，fswatch 可捕获；在某些 overlay 配置下可能无法捕获。必须在目标环境实测验证。

风险编号风险描述严重性缓解策略状态
RISK-01 SIGSTOP 挂起期间 Claude Code 与 Anthropic 服务器的 TCP 长连接超时（通常 60-120s） 高 Gateway 恢复前检查连接状态；断线则重新注入 current_spec.md 上下文片段；建议部署 Ollama 本地备选模型 ⚠️ 待实测
